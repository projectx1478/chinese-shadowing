# 保留事項・既知の問題

- Gemini音声合成（より自然なTTS）の導入は見送り中（無料枠のRPD制限のため）。将来的に制約が緩和されれば再検討の余地あり
- OSのTTS音声設定画面へアプリから直接遷移させる手段はない（Web標準APIでは非対応、iOS/Android共通）。案内文言での対応は今回見送り
- 声調概形のオーバーレイは音節の時間割り当てが均等割りの近似であり、精度は粗め（仕様として許容済み）
- 状態色（成功/警告/危険）とHSKレベル別の色は、テーマ変数化せず固定のままにしている（どちらのテーマでも視認性が保てるための意図的な設計）
- 発音解説・フィードバック等の一部ボックスは内部で完結した配色（テーマに関わらず固定）としている
- AIコーチの「会話練習」枠は現状シャドーイングモードと同一実装であり、専用の会話継続ロジックは未分離（Phase2で検討）
- Cloudflare Worker（`worker.js`）はGitHubリポジトリに含めていないため、Worker側の変更履歴はPROJECT.mdでのみ管理している点に注意
- `app.js` はさらなる機能別分割（config/firebase/gemini/audio/practice等）を将来的に検討可能だが、巻き上げ依存の読み込み順序リスクがあるため、着手時は依存関係を精査してから段階的に行うこと
- Playwrightスモークテストは認証・録音・実機TTSを対象外としているため、これらの最終確認は引き続き実機（iPad Pro / AQUOS9）で行う必要がある
- AI解析（録音後）の速度改善は、採点の一貫性とのトレードオフがあるため未着手（上記「次回最優先作業」参照）
- ミニマルペア聞き分け（8-3）は「同じ/違う」判定形式は未実装（2択のみ）。`MINIMAL_PAIRS`は26ペアの初期セットのみで、将来的に拡充の余地あり
- 穴埋めディクテーション（8-1）・ミニマルペア（8-3）・内容理解クイズ（8-2）とも、`updateStats()`の集計ロジックが`type!=="dictation"`を全て「🎙シャドーイング」統計として扱うため、トップの集計バッジでは新モードの実績がシャドーイング扱いで混ざって表示される（履歴一覧では専用バッジで区別可能）。実害は軽微だが、モード種別ごとの集計を厳密にするなら`updateStats()`の見直しが必要
- 内容理解クイズ（8-2）の質問の型（誰が何をしたか／いつ・どこで／感情・意図／次に起こること）はプロンプトで例示しているのみで、AIが必ずしも毎回異なる型を選ぶ保証はない（プロンプト任せ）
- バックワードビルドアップ（8-4）は録音・採点を既存のシャドーイング用パイプラインをそのまま流用しているため、履歴に専用の`type`が付かず（`shadow`/`full`と同様に無印）、履歴一覧・統計上は通常のシャドーイングと区別が付かない（8-5の設計方針通りの意図的な挙動だが、区別したい場合は別途対応が必要）
- PROJECT.md 8章の4モードは4件ともUI・ロジックレベルではPlaywrightで検証済みだが、実際のGemini API呼び出し（プロンプトから意図通りのタグが返るか）・実機でのWeb Speech API音声品質は未検証。特に8-4のAI生成チャンク分割が「意味のまとまり」として自然かどうかは実際の生成結果を見るまで不明
- 8-6作文練習の音声入力（`webkitSpeechRecognition`）はSafari（iOS）・Chrome（Android）を想定しているが、実機での認識精度（特に中国語入力）は未検証。非対応ブラウザでは「音声入力に対応していません」とアラートを出すのみでテキスト入力は引き続き利用可能
